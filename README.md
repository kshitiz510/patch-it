## Quick Start

### 1. Backend (Express API)

```bash
cd backend
npm install
cp .env.example .env          # edit MONGO_URI if needed
npm run dev                   # → http://localhost:4000
```

Requires MongoDB running locally (`mongodb://localhost:27017/patchit` by default).

### 2. Frontend (React SPA)

```bash
cd frontend
npm install
npm run dev                   # → http://localhost:5173
```

Create `frontend/.env` if your backend isn't on port 4000:

```env
VITE_API_URL=http://localhost:4000
```

### 3. Blockchain (Tender Bidding)

```bash
cd blockchain
npm install

# Start local Hardhat node (dev chain):
npx hardhat node

# In another terminal, deploy & test:
npx hardhat test                          # Run tests
npx hardhat run scripts/deploy.js         # Deploy to localhost
```

> **MetaMask Setup:** Point MetaMask to Hardhat's local chain (RPC `http://127.0.0.1:8545`, Chain ID `31337`). Import a Hardhat account's private key to test transactions.

### 4. ML Service (Django + YOLO)

```bash
cd ml
python -m venv .venv

# Windows
.venv\Scripts\activate

# macOS / Linux
source .venv/bin/activate

pip install -r requirements.txt

# Install PyTorch for YOUR CUDA version:
# https://pytorch.org/get-started/locally/
# Example (CUDA 12.1):
# pip install torch torchvision --index-url https://download.pytorch.org/whl/cu121

python manage.py migrate
python manage.py runserver 8000   # → http://localhost:8000
```

See [ml/README.md](ml/README.md) for dataset sources and training instructions.

## ML Pipeline (End-to-End)

### Step 1 — Download Training Data

You need **YOLO-format** pothole detection data. Recommended sources:

| Source                   | Link                                                                                      | Notes                           |
| ------------------------ | ----------------------------------------------------------------------------------------- | ------------------------------- |
| Roboflow Pothole Dataset | [roboflow.com/universe/pothole-detection](https://universe.roboflow.com/search?q=pothole) | Ready YOLO format, 5000+ images |
| Kaggle Pothole Dataset   | [kaggle.com/datasets](https://www.kaggle.com/search?q=pothole+detection)                  | Convert to YOLO format          |
| CRDDC2022                | [github.com/sekilab/RoadDamageDetector](https://github.com/sekilab/RoadDamageDetector)    | Multiple damage types           |

Organize into this structure:

```
ml/
└── data/
    ├── data.yaml          # auto-generated by train.py
    ├── train/
    │   ├── images/        # training images (.jpg, .png)
    │   └── labels/        # YOLO format .txt files
    └── val/
        ├── images/        # validation images
        └── labels/        # validation labels
```

**YOLO label format** (one `.txt` per image, same filename):

```
<class_id> <x_center> <y_center> <width> <height>
```

For single-class (pothole): class_id is always `0`. All coordinates are normalized (0–1).

### Step 2 — Preprocess (Optional)

```bash
cd ml
python scripts/preprocess.py
```

Applies:

- Resize to 640×640
- CLAHE contrast enhancement
- Data augmentation (flips, rotations, brightness via Albumentations)

### Step 3 — Train

```bash
cd ml
python scripts/train.py
```

This will:

1. Auto-generate `data/data.yaml`
2. Load `yolov8n` pretrained weights
3. Train for 50 epochs (batch=16, imgsz=640)
4. Save the best model to `ml/models/pothole_yolo/weights/best.pt`

**GPU recommended.** On CPU, training takes ~4-6 hours. On a 3060+, ~30 minutes.

Monitor training with TensorBoard:

```bash
tensorboard --logdir ml/models/pothole_yolo
```

### Step 4 — Evaluate

```bash
python scripts/evaluate.py
```

Outputs precision/recall/mAP50 metrics and confusion matrices.

### Step 5 — Single Image Inference

```bash
python scripts/predict.py --image data/test/sample.jpg
```

Opens a matplotlib window with bounding boxes and confidence scores.

### Step 6 — Video Inference (Real-time)

```bash
python scripts/video_inference.py --input data/video.mp4 --output data/output.mp4
```

Processes each frame through YOLO, outputs annotated video with bounding boxes.

### Step 7 — Serve via API (Real-time)

Start the Django server:

```bash
python manage.py runserver 8000
```

Then send images for detection:

```bash
curl -X POST http://localhost:8000/api/detect/ \
  -F "image=@path/to/pothole.jpg"
```

Response:

```json
{
  "detections": [
    {
      "confidence": 0.92,
      "bbox": [120.5, 80.3, 340.2, 290.1],
      "class": "pothole",
      "severity": "high"
    }
  ],
  "count": 1
}
```

Severity classification:

- **High:** confidence ≥ 0.8
- **Medium:** 0.5 ≤ confidence < 0.8
- **Low:** confidence < 0.5

## Bidding Process Explained

The tender bidding system uses an **Ethereum smart contract** (`TenderBidding.sol`) for fully transparent, tamper-proof road repair procurement.

### How It Works

```
Admin creates tender → Bidders register → Bidders place bids → Admin closes tender → Winner selected
```

### Roles

| Role       | Who                                | Actions                                        |
| ---------- | ---------------------------------- | ---------------------------------------------- |
| **Admin**  | Municipality / government official | Deploy contract, create tenders, close tenders |
| **Bidder** | Contractors / repair companies     | Register, place bids on open tenders           |

### Detailed Flow

1. **Deploy Contract:** Admin deploys `TenderBidding.sol` to the network (or loads existing address). The deployer calls `createAdmin(name)` to register as admin.

2. **Register Bidders:** Each contractor calls `registerBidder(name)` with their wallet. This maps their Ethereum address to their name.

3. **Create Tender:** Admin calls `createTender(latitude, longitude, baseAmount)`.
   - Latitude + longitude concatenate into a unique key (`"28.6139_77.2090"`)
   - `baseAmount` is the government's budget estimate in wei
   - Duplicate tender keys are rejected

4. **Place Bids:** Registered bidders call `placeBid(tenderKey, amount)`.
   - Bids **cannot exceed 110%** of the base amount (prevents inflated bids)
   - Multiple bids per bidder are allowed
   - Bids are sorted by amount (lowest first) on-chain

5. **Close Tender:** Admin calls `closeTender(tenderKey)`.
   - Tender is marked as closed
   - The bidder with the **lowest bid** is automatically selected as winner
   - Winner address and bid amount are recorded permanently

6. **View Results:** Anyone can call:
   - `listOpenTenders()` → returns keys[] and baseAmounts[]
   - `listClosedTenders()` → returns keys[], winnerNames[], and amounts[]

### Key Smart Contract Rules

- **110% cap:** No bid can exceed 110% of the base amount
- **Lowest wins:** Bids are bubble-sorted on-chain; lowest amount wins
- **Admin-only control:** Only admin accounts can create/close tenders
- **Registration required:** Must call `registerBidder()` before placing bids
- **Permanent record:** All transactions are on-chain and verifiable
- **Unique tenders:** Each lat/lng combination can only have one tender

### Using the Bid Page

1. Install MetaMask and connect to Hardhat localhost (`http://127.0.0.1:8545`, Chain ID `31337`)
2. Import a Hardhat test account (copy private key from `npx hardhat node` output)
3. Start Hardhat node: `cd blockchain && npx hardhat node`
4. Deploy contract: `npx hardhat run scripts/deploy.js --network localhost`
5. On the Bid page: **Connect Wallet → Deploy (or Load) → Register → Create/Bid**
6. Contract address persists in localStorage across sessions

## Feature Suggestions

Ideas for expanding Patch-It into a more complete platform:

### High Priority

- **Automatic pothole detection from uploaded videos** — Run YOLO on uploaded videos server-side, extract frames with detections, and auto-create reports with annotated images.
- **Severity scoring & priority queue** — Use detection confidence + depth estimation to assign priority levels (critical/major/minor). Municipalities see highest-priority repairs first.
- **User authentication** — Add JWT-based auth so users can track their reports, see status updates, and build reputation scores.
- **Push notifications** — Notify reporters when their reported pothole gets a tender created or repaired.

### Medium Priority

- **Before/After verification** — After repair, users re-upload footage of the same location. AI compares frames to verify the fix.
- **Mobile-first PWA** — Add service workers, offline support, and installability for field workers and drivers.
- **Ward/zone analytics** — Auto-assign reports to municipal wards using reverse geocoding. Dashboard shows per-ward stats, trends, and response times.
- **Upvote/confirm system** — Multiple citizens can confirm the same pothole, increasing its priority weight. Prevents false reports.
- **Tender escrow** — Lock bid amounts in the smart contract and release after repair verification, adding financial accountability.

### Nice to Have

- **Dashcam integration** — Real-time detection while driving with auto-upload when reaching Wi-Fi.
- **Government dashboard** — Admin portal with geographic analytics, repair budget allocation, contractor performance metrics.
- **Cost estimation ML** — Second model that estimates repair cost based on pothole dimensions, depth, and road type.
- **Multi-language support** — i18n for Hindi, Tamil, Bengali, etc. for broader adoption.
- **Gamification** — Leaderboards for most active reporters by city, monthly badges, achievement system.
- **Contractor rating system** — Post-repair quality ratings that affect future tender eligibility.
